# Overview

## What This Tool Does

This tool quantifies AC-side losses between a site's inverter output and its production meter. It ingests raw time-series data from a data acquisition system (DAS), applies a multi-stage cleaning pass to remove noise and non-productive periods, computes efficiency and loss metrics, and produces structured reports for analysis.

The goal is to answer a simple but operationally important question: **how much power generated by the inverters is actually making it through to the meter, and when are losses largest?**

---

## The Problem It Solves

In a solar AC system, the inverters are the last point of DC-to-AC conversion before power flows to the utility interconnect. The production meter sits downstream. Any discrepancy between the two — after accounting for expected wiring and transformer losses — may indicate:

- Inverter underperformance relative to nameplate
- Meter drift or calibration issues
- Partial curtailment events not captured in SCADA
- Wiring or breaker-level losses not otherwise instrumented

Most asset management platforms surface aggregate MWh-level discrepancies. This tool works at the row level (typically 1- or 15-minute intervals), which makes it possible to identify **when** losses are occurring (time of day, month, operating regime) rather than just how large they are in aggregate.

---

## Signals Analyzed

The tool works with the following measurement streams from the raw DAS export:

- **Inverter AC output power** — per-inverter active power readings, used as the "supply" side of the efficiency calculation
- **Three-phase AC currents** — phase A, B, and C current readings used to detect electrical imbalance conditions that indicate faulty or anomalous operating states
- **Three-phase AC voltages** — phase A, B, and C voltage readings available for future analysis (currently loaded but not used in cleaning or calculations)
- **Production meter reading** — the site-level active power measurement at the point of interconnection, used as the "delivered" side of the efficiency calculation
- **Timestamp** — the local site time for each measurement row, used to filter periods and assign time-of-day buckets

---

## The Four-Stage Pipeline

```
LOAD  →  CLEAN  →  CALCULATE  →  REPORT
```

| Stage | What Happens |
|---|---|
| **Load** | Discovers all CSVs in the input folder, parses timestamps and numeric columns, reports a row count and date range per site |
| **Clean** | Applies four sequential filters to remove nighttime rows, offline readings, phase-imbalance anomalies, and gross efficiency outliers |
| **Calculate** | Derives efficiency percentage, loss delta, and time-of-day classification for every remaining row |
| **Report** | Writes a per-site cleaned CSV and a single multi-tab Excel workbook covering all sites |

Each stage is independent and self-contained. The output of one stage feeds the next through a shared `SiteRecord` object that carries the DataFrame at each transformation step.

---

## What a Typical Run Looks Like

1. You drop one or more site CSV exports into `data/raw/`
2. You run `main.py`
3. The console prints per-stage progress: file discovery, row counts before and after each filter, efficiency averages, and output file paths
4. In `output/`, you find a cleaned CSV per site and one Excel workbook covering all sites

Total runtime for a single year of 15-minute data from a two-inverter site is a few seconds. The Excel workbook opens automatically on Windows when the run completes.

---

## What This Tool Is Not

- It is not a real-time monitoring system — it operates on exported batch files
- It does not diagnose the root cause of losses — it surfaces where and when to look
- It does not ingest irradiance or performance ratio data — efficiency is computed purely from the electrical measurements present in the DAS export
